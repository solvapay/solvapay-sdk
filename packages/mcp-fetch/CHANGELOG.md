# @solvapay/mcp-fetch changelog

From `0.1.1` onwards this changelog is generated by
[changesets](https://github.com/changesets/changesets) — entries below
the inaugural release are maintained by hand.

## 0.1.0

Inaugural release. Fetch-first OAuth bridge + turnkey MCP handler for
Web-standards runtimes: Deno, Supabase Edge, Cloudflare Workers, Bun,
Next.js Edge, Vercel Functions.

### Turnkey handler

`createSolvaPayMcpFetchHandler({ server, publicBaseUrl, apiBaseUrl, productRef, mcpPath? })`
composes the OAuth bridge, a
`WebStandardStreamableHTTPServerTransport`-backed MCP endpoint, and
native-scheme CORS into a single `(req: Request) => Promise<Response>`
— drop it into `Deno.serve`, a Cloudflare Worker `fetch` handler,
`Bun.serve`, a Next.js Edge route, or any other Web-standards runtime
and you have a fully bootstrapped SolvaPay MCP server.

### Individual fetch handlers

For integrators who want to own their router:

- `createOAuthProtectedResourceFetchHandler`
- `createOAuthAuthorizationServerFetchHandler`
- `createOAuthRegisterFetchHandler`
- `createOAuthAuthorizeFetchHandler`
- `createOAuthTokenFetchHandler`
- `createOAuthRevokeFetchHandler`

Each returns `(req: Request) => Promise<Response | null>` — `null` if
the request doesn't match the handler's method + path, so they compose
cleanly via a ladder of short-circuit `?? await next(req)` calls.

### Primitives

- `resolveBearer(req)` — extract the `Authorization: Bearer …` token.
- `respondUnauthorized({ realm, scope?, error?, errorDescription? })` —
  return a `401` with the RFC 9728 `WWW-Authenticate` challenge shape.
- Native-scheme CORS helpers that mirror `Access-Control-Allow-Origin`
  for `cursor://`, `vscode://`, `vscode-webview://`, and `claude://`
  clients — matches `@solvapay/mcp-express`'s behaviour.
- Byte-for-byte body passthrough on `/oauth/token` and
  `/oauth/revoke` (handles `+` vs `%20` encoding quirks correctly on
  the `Request` stream).

### Contracts

Peers on `@solvapay/mcp-core`, `@solvapay/mcp`, and
`@modelcontextprotocol/sdk ^1.29`. RFC 8414 §3.3 self-consistency is
asserted in the test suite — the `authorization_server` response's
`issuer` always equals the configured `publicBaseUrl`, and no
SolvaPay-internal fields (`product_ref`, upstream URLs) leak into the
discovery JSON.
