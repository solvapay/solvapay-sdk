# @solvapay/fetch changelog

From `1.0.1` onwards this changelog is generated by
[changesets](https://github.com/changesets/changesets) — entries below
the rename release are maintained by hand.

## 1.0.0

Renamed from `@solvapay/supabase` → `@solvapay/fetch`. Code carried
over verbatim from `@solvapay/supabase@1.0.1` — every public symbol
keeps its name, signature, and behaviour. Migration is a single find
and replace:

```diff
- import { checkPurchase, createPaymentIntent } from '@solvapay/supabase'
+ import { checkPurchase, createPaymentIntent } from '@solvapay/fetch'
```

```diff
- "@solvapay/supabase": "^1.0.1"
+ "@solvapay/fetch": "^1.0.0"
```

Exports unchanged: `checkPurchase`, `createPaymentIntent`,
`trackUsage`, `activatePlan`, `cancelRenewal`, `reactivateRenewal`,
`createCheckoutSession`, `createCustomerSession`,
`createTopupPaymentIntent`, `customerBalance`, `getMerchant`,
`getProduct`, `getPaymentMethod`, `listPlans`, `processPayment`,
`solvapayWebhook`, `syncCustomer`, `corsHeaders`, plus the usual type
exports.

### Why rename?

The package body is runtime-neutral — fetch-first handlers around
`@solvapay/server` cores. Nothing Supabase-specific lives in the code.
The Supabase-idiomatic examples continue to ship under
[`examples/supabase-edge/`](../../examples/supabase-edge/) (and now
[`examples/supabase-edge-mcp/`](../../examples/supabase-edge-mcp/) for
the full MCP server). The old name was creating false discovery
friction for Cloudflare Workers / Bun / standalone Deno / Next.js
Edge / Vercel Functions integrators who would otherwise have been the
natural audience for the package.

### No deprecation shim

`@solvapay/supabase` stays in the npm registry at `1.0.1` (`@latest`)
and `1.0.8-preview.*` (`@preview`) with no new releases. A hard rename
across one releases window is cleaner than a permanently dual-named
package. Existing `@solvapay/supabase` installs keep working — they
just stop receiving updates.
